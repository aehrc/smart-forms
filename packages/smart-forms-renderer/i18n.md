# Internationalization (i18n) in the renderer

Status: **proposal / first slice** ŌĆö see issue [#1996](https://github.com/aehrc/smart-forms/issues/1996) (motivation) and draft RFC PR [#1995](https://github.com/aehrc/smart-forms/pull/1995) (discussion). This document describes the approach so we can align on it and continue with the next steps.

## Goal

Let a consuming app localize the renderer's **own** output (labels, messages, date formatting) per language/region, opt-in and backward-compatible. English / `DD/MM/YYYY` remain the default when nothing is configured.

## Two layers of i18n

There are two distinct concerns. They have different sources of truth and are solved differently.

| Layer | What | Source of truth | Mechanism |
|-------|------|-----------------|-----------|
| **1. Renderer chrome** | Text the renderer owns: Yes/No, validation messages, buttons, aria-labels; plus date formatting | The renderer's code | A locale-driven string catalog + `Intl` for dates |
| **2. Questionnaire content** | `item.text`, `answerOption.display`, group/questionnaire titles | The `Questionnaire` resource | FHIR `Questionnaire.language` + the `translation` extension on `_text` |

**This slice implements Layer 1 only** (and only a first set of strings). Layer 2 is a separate, larger follow-up.

## Architecture (Layer 1)

Everything is configured through the existing `rendererConfigStore`, via two new fields on `RendererConfig`:

- `locale?: string` ŌĆö a BCP-47 tag (e.g. `'de-CH'`). Selects a bundled catalog and drives date formatting.
- `rendererStrings?: Partial<RendererStrings>` ŌĆö an optional per-string override, merged on top of the locale catalog.

Set via `buildForm({ rendererConfigOptions: { locale, rendererStrings } })`.

### String catalog

- `RendererStrings` (`src/i18n/rendererStrings.ts`) ŌĆö the typed contract: every renderer-owned string is a key here.
- `defaultRendererStrings` ŌĆö English defaults; the base every locale/override merges onto. Must contain every key.
- `src/i18n/locales/*.json` ŌĆö bundled locale catalogs authored as **JSON translation files** (`de-CH`, `fr-CH`, `it-CH`). Only keys that differ from English need to be present.
- `bundledRendererStrings` ŌĆö registry mapping locale tag ŌåÆ catalog.
- `resolveRendererStrings(locale, overrides)` ŌĆö merge order: English defaults ŌåÉ locale catalog ŌåÉ explicit overrides. Region tags fall back to base language, unknown locales fall back to English.
- `interpolate(template, params)` ŌĆö replaces `{token}` placeholders at runtime (for messages with dynamic values).

Consumed in components via `useRendererConfigStore.use.rendererStrings()`.

### Dates

The date format is **derived from the locale**, not stored per locale.

- `resolveDateFormat(locale, override)` (`.../DateTimeItems/utils/parseDate.ts`):
  1. explicit `dateFormat` override, else
  2. derived from the locale via `Intl.DateTimeFormat().formatToParts()` (all locales, no bundled data: `de-CH`ŌåÆ`DD.MM.YYYY`, `en-US`ŌåÆ`MM/DD/YYYY`, `ja-JP`ŌåÆ`YYYY/MM/DD`), else
  3. `DD/MM/YYYY` fallback.
- `useDateFormat()` ŌĆö reactive hook wrapping the resolver for components.
- Validation (`validateThreeMatches`, `useDateValidation`) is **order-aware**: positional input parts are mapped to day/month/year using the format's token order, so day-first, month-first (US) and year-first all validate correctly.
- Validation error messages are catalog templates with `{format}` / `{separator}` / `{monthYearFormat}` placeholders, interpolated at runtime.
- The renderer keeps its **custom parser** (rather than `Intl`/MUI-native fields) because it supports FHIR **partial dates** (`YYYY`, `YYYY-MM`).

### Calendar popup

The date-picker calendar's month/weekday names come from `dayjs` locale data. `BaseRenderer` imports `dayjs/locale/{de-ch,fr-ch,it-ch}` and passes `adapterLocale` to `LocalizationProvider`. The **field format needs no dayjs locale import** (that's Intl); these imports are only for the calendar popup. Consumers wanting a localized calendar for other locales import the matching `dayjs/locale/<tag>` themselves.

## Usage (consumers)

```ts
// Locale drives labels, date format, and validation messages together
buildForm({ questionnaire, rendererConfigOptions: { locale: 'de-CH' } });
// ŌåÆ Ja / Nein, dates DD.MM.YYYY, German validation errors

// Override individual strings / the date format, independent of locale
buildForm({ questionnaire, rendererConfigOptions: {
  rendererStrings: { booleanYesLabel: 'Oui', dateFormat: 'MM/DD/YYYY' }
}});
```

The library imposes no i18n framework: an app already using i18next/FormatJS/etc. loads its own strings however it likes and passes a plain `Partial<RendererStrings>` object.

## Contributing

**Add a bundled locale**
1. Create `src/i18n/locales/<tag>.json` with the translated keys (only those differing from English).
2. Register it in `bundledRendererStrings` in `src/i18n/rendererStrings.ts`.
3. (Optional) import `dayjs/locale/<tag>` in `BaseRenderer` for calendar localization.
4. Tests enforce: no unknown keys, and placeholders preserved vs the English default.

**Add a new renderer string**
1. Add the key to the `RendererStrings` interface with a doc comment.
2. Add the English value to `defaultRendererStrings`.
3. Add the key to `validRendererStringKeys` in `src/test/rendererStrings.test.ts` (typed `Record<keyof RendererStrings, true>`, so this is compiler-enforced).
4. Replace the hardcoded literal in the component/hook with a catalog lookup (`interpolate(...)` if it has placeholders).
5. Add translations to the locale JSON files as needed.

## Key design decisions (reasoning)

- **Config-injected catalog, not an i18n framework** ŌĆö the renderer is a published library; a framework would force peer deps/a provider/a workflow on every consumer. Defaults + optional bundled locales + object injection keeps the common case zero-setup.
- **JSON locale files (not `.ts`/`.po`)** ŌĆö translator-tool friendly (Crowdin/Weblate/Poedit), no runtime parser, `tsc` emits them to `lib/`. Type safety kept via the `Record` registry + a key guard + a placeholder guard. `.po`/Lingui (best translator tooling) needs a compile step ŌĆö revisit if a translation-management workflow is wanted.
- **Dates via `Intl`, not hardcoded/per-locale imports** ŌĆö all locales for free, no bundled data, no import list; explicit override remains.
- **Order-aware validation** ŌĆö prerequisite for locale-driven formats (US month-first, etc.).
- **Interpolated messages** ŌĆö translations stay natural while dynamic values are injected; a test guards placeholder consistency.
